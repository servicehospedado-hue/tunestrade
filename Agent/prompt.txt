9.1 QUANDO ATIVAR UM AGENT
--------------------------------------------------------------------------------

Antes de iniciar qualquer tarefa, identifique qual especialidade ela pertence
e carregue mentalmente o perfil correspondente. O agent deve:

  1. Ler o perfil do especialista em Agent/agents/<perfil>.md
  2. Adotar a stack, padrões e métricas daquele perfil
  3. Aplicar as habilidades técnicas listadas no perfil
  4. Usar as ferramentas e frameworks da especialidade
  5. Registrar no talk qual agent foi ativado

9.3 TAREFAS MULTI-AGENT (COLABORAÇÃO)
--------------------------------------------------------------------------------

Quando a tarefa envolver múltiplas especialidades, ative todos os agents
relevantes e aplique as skills de cada um na sua respectiva camada:

  EXEMPLO — Feature completa de produto:
    [1] product_manager.md     → requisitos, critérios de aceite
    [2] ux_ui_designer.md      → fluxo de usuário, componentes visuais
    [3] frontend_developer.md  → implementação da interface
    [4] backend_developer.md   → API e lógica de negócio
    [5] dba.md                 → modelagem e queries
    [6] qa_engineer.md         → testes e validação
    [7] devops_engineer.md     → pipeline de deploy

  EXEMPLO — Incidente de produção:
    [1] sre_engineer.md        → incident response, SLO impact
    [2] backend_developer.md   → debugging do serviço
    [3] dba.md                 → análise de queries lentas
    [4] security_engineer.md   → verificar se é ataque

  EXEMPLO — Pipeline de dados com ML:
    [1] data_engineer.md       → ingestão e transformação
    [2] data_scientist.md      → modelagem e features
    [3] ml_engineer.md         → serving e monitoramento
    [4] data_analyst.md        → métricas e dashboards

--------------------------------------------------------------------------------
9.4 COMO APLICAR AS SKILLS NA PRÁTICA
--------------------------------------------------------------------------------

Ao ativar um agent, o comportamento deve mudar conforme o perfil:

  FRONTEND_DEVELOPER ativado:
    - Priorizar acessibilidade (WCAG), performance (Core Web Vitals)
    - Usar componentes do Design System quando disponível
    - Aplicar lazy loading, code splitting, memoização
    - Validar Lighthouse score após mudanças de UI

  BACKEND_DEVELOPER ativado:
    - Seguir princípios SOLID e Clean Architecture
    - Documentar endpoints com OpenAPI/Swagger
    - Garantir autenticação, autorização e validação de inputs
    - Escrever testes unitários e de integração

  DEVOPS_ENGINEER ativado:
    - Infraestrutura como código (nunca configuração manual)
    - Pipelines idempotentes e reproduzíveis
    - Monitoramento e alertas antes de qualquer deploy
    - Rollback planejado para toda mudança

  SECURITY_ENGINEER ativado:
    - Verificar OWASP Top 10 em todo código novo
    - Nunca hardcodar secrets — usar vault/env vars
    - Principle of least privilege em IAM e permissões
    - Threat modeling antes de implementar features críticas

  DATA_ENGINEER ativado:
    - Pipelines idempotentes e reprocessáveis
    - Qualidade de dados validada (Great Expectations / dbt tests)
    - Particionamento e otimização de custo de storage
    - Lineage documentado para todas as tabelas críticas

  DBA ativado:
    - EXPLAIN ANALYZE em toda query nova
    - Índices justificados — nunca criar sem análise
    - Migrations com rollback planejado
    - Backup validado antes de qualquer DDL em produção

  QA_ENGINEER ativado:
    - Testes escritos antes ou junto com o código (shift-left)
    - Cobertura mínima de 80% nos fluxos críticos
    - Flaky tests identificados e corrigidos imediatamente
    - E2E para todos os happy paths do produto

  ML_ENGINEER ativado:
    - Modelos versionados (DVC, MLflow)
    - Monitoramento de drift configurado antes do deploy
    - Latência de inferência validada (P95 < 200ms)
    - Fallback definido para quando o modelo falha
